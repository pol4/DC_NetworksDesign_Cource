# Построение Underlay сети (OSPF + BFD) в архитектуре Leaf-Spine

Этот репозиторий содержит конфигурационные файлы и документацию для построения Underlay-сети с использованием протокола OSPF и BFD в архитектуре Leaf-Spine.

## 1. Топология сети

Архитектура состоит из 2-х Spine коммутаторов и 3-х Leaf коммутаторов.
- К `leaf1` и `leaf2` подключено по одному серверу.
- К `leaf3` подключено два сервера.

## 3. Логика работы OSPF и BFD

### 3.1. OSPF (Open Shortest Path First)

Протокол OSPF используется для построения Underlay-сети, обеспечивая полную IP-связность между всеми коммутаторами (Leaf и Spine). Это является фундаментом для дальнейшего построения оверлейных сервисов (например, VXLAN).

- **`router ospf 1`**: Запускает процесс OSPF с номером процесса 1.
- **`router-id`**: В качестве ID роутера используется IP-адрес с интерфейса `Loopback0`. Это обеспечивает стабильность `router-id`, так как loopback-интерфейсы всегда активны.
- **`area 0.0.0.0`**: Все интерфейсы включены в единую магистральную зону OSPF (backbone area). В данной топологии нет необходимости в разделении на несколько зон.
- **`ip ospf network point-to-point`**: Эта команда используется на всех линках между Leaf и Spine. Она оптимизирует работу OSPF:
    - Ускоряет установление соседства, так как не требуется выбор DR/BDR.
    - Экономит IP-адреса, позволяя использовать маски `/31`.
- **`passive-interface default`**: Команда по умолчанию делает все интерфейсы пассивными. Это означает, что по умолчанию ни на одном интерфейсе не будут отправляться OSPF Hello-пакеты и, соответственно, не будут устанавливаться отношения соседства.
- **`no passive-interface <interface>`**: Эта команда выборочно активирует OSPF на нужных интерфейсах:
    - На физических интерфейсах (`Ethernet1`, `Ethernet2` и т.д.), смотрящих в сторону других коммутаторов, чтобы установить с ними соседство.
    - На интерфейсах VLAN (`Vlan10`, `Vlan20`, `Vlan30`), чтобы анонсировать подсети серверов в OSPF-домен. При этом соседство на этих интерфейсах устанавливаться не будет, так как на другой стороне находятся серверы, не участвующие в OSPF.

### 3.2. BFD (Bidirectional Forwarding Detection)

BFD используется для сверхбыстрого обнаружения сбоев на пути между двумя маршрутизаторами. В данной конфигурации BFD интегрирован с OSPF для ускорения сходимости сети.

- **`bfd interval 300 min-rx 300 multiplier 3`**: Эта команда настраивается на каждом физическом интерфейсе между Leaf и Spine.
    - `interval 300`: BFD-пакеты отправляются каждые 300 миллисекунд.
    - `min-rx 300`: Ожидаемый минимальный интервал приема BFD-пакетов от соседа - 300 мс.
    - `multiplier 3`: Если 3 последовательных BFD-пакета не будут получены от соседа, сессия BFD будет считаться разорванной.
    - **Итоговое время обнаружения сбоя:** 300 мс * 3 = 900 мс. Это значительно быстрее, чем стандартные таймеры OSPF (40 секунд).
- **`bfd all-interfaces`**: Эта команда в секции `router ospf 1` автоматически включает интеграцию BFD со всеми интерфейсами, на которых активен OSPF. Как только BFD обнаруживает сбой на линке, он немедленно уведомляет OSPF, который, в свою очередь, мгновенно начинает пересчет маршрутов, не дожидаясь истечения своих таймеров.

## 5. Порядок развертывания и проверки

1.  Загрузить соответствующую конфигурацию на каждое устройство.
2.  Применить сетевые настройки на серверах в соответствии с таблицей в п. 2.3.
3.  **Проверка BFD и OSPF:**
    - На каждом коммутаторе выполнить команду `show bfd neighbors`, чтобы убедиться, что BFD-сессии установлены (`Status: Up`).
    - На каждом коммутаторе выполнить команду `show ip ospf neighbor`, чтобы убедиться, что установлены отношения соседства со всеми подключенными коммутаторами.
    - На любом коммутаторе выполнить `show ip route ospf`, чтобы проверить, что в таблице маршрутизации присутствуют маршруты ко всем `Loopback` интерфейсам и `VLAN` интерфейсам других коммутаторов.
4.  **Проверка связности:**
    - С каждого коммутатора выполнить `ping` до `Loopback` IP-адресов всех остальных коммутаторов в сети.
    - С сервера `server1` (`172.16.10.1`) выполнить `ping` до своего шлюза `172.16.10.254`.

## 6. Проверка ECMP (Equal-Cost Multi-Path)

ECMP позволяет распределять трафик по нескольким путям с одинаковой стоимостью. В нашей топологии у каждого Leaf-коммутатора есть два равноценных пути до любой удаленной подсети через `spine1` и `spine2`.

Наличие ECMP можно проверить командой `show ip route` на любом из Leaf-коммутаторов.

### Анализ вывода:

Ключевые моменты, подтверждающие работу ECMP:

-   **Маршрут к `172.16.20.0/24` (сеть за `leaf2`):**
    ```
     O        172.16.20.0/24 [110/30]
               via 10.0.0.0, Ethernet1
               via 10.0.0.6, Ethernet2
    ```
    Здесь видно два `next-hop` адреса для одного и того же маршрута: один через `spine1` (на `10.0.0.0`), другой через `spine2` (на `10.0.0.6`).

-   **Маршрут к `192.168.0.103/32` (Loopback `leaf3`):**
    ```
     O        192.168.0.103/32 [110/30]
               via 10.0.0.0, Ethernet1
               via 10.0.0.6, Ethernet2
    ```
    Аналогично, для достижения `Loopback` интерфейса `leaf3` также установлено два равноценных маршрута.

Это доказывает, что `leaf1` будет использовать оба своих аплинка к `spine1` и `spine2` для отправки трафика к удаленным сетям, эффективно распределяя нагрузку.
